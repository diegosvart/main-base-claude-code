# Executive Summary: main-base-claude-code v1.0.0

> A Claude Code project template that turns a vague project idea into a validated technical blueprint before the first line of code is written.

**The problem**: Starting a Claude Code project without structure = context scattered across conversations, quality inconsistent, decisions made late, work often restarted.

**The solution**: 6 pre-configured layers + an inception agent that guides 30-60 minute structured discovery, producing REQUIREMENTS.md, ARCHITECTURE.md, PLAN.md, and CLAUDE.md before bootstrap.

**Release date**: 2026-03-18 | **Stack**: Python/FastAPI (base) + extensible to any stack | **License**: MIT

---

## What Does It Do? (30-second version)

The template is a **discovery-first scaffold** designed for teams using Claude Code. It provides:

1. **Inception Agent** — runs a 5-phase discovery protocol (problem → scope → tech stack → dev plan) that surfaces decisions early
2. **6 pre-configured layers** — memory (CLAUDE.md), skills, commands, hooks, MCP, agents — ready to use immediately
3. **Quality gates** — automatic linting, formatting, tests on every write; session audit log; phase summaries on merge
4. **5 specialized agents** — TL/Orchestrator (always active) + Architecture + Backend + DBA + Frontend (load per stack)
5. **Session recovery** — works with [claude-mem](https://github.com/thedotmack/claude-mem) to resume interrupted work from exact state

**Outcome**: a project that starts *validated*, not discovered-as-you-go.

## 6-Layer Architecture

Every project cloned from this template includes:

| Layer | Location | Function |
|---|---|---|
| **1. Memory** | `CLAUDE.md` | Project context — stack, conventions, agents mounted. Generated by inception or filled manually. |
| **2. Skills** | `.claude/skills/` | Playbooks installed per stack (Python, Node, etc.) — domain-specific workflows. |
| **3. Commands** | `.claude/commands/` | Slash commands: `/commit`, `/new-feature`, `/create-pr`, `/review`, `/close-phase`. |
| **4. Hooks** | `.claude/settings.json` | Quality gates: auto-format on write, session log, test runs, audit trail. |
| **5. MCP** | `.mcp.json` | External servers (empty by default; inception configures per stack). |
| **6. Agents** | `.claude/agents/` | 5 specialized agents (TL + Architecture + Backend + DBA + Frontend). |

Each layer can be adopted independently. You don't need all 6 to get value — hooks alone enforce quality, agents alone provide guidance.

## The Inception Agent: 3 Modes

The agent orchestrates discovery via 3 entry points:

```bash
/inception "describe your project idea"    # Mode: fresh discovery
/inception --resume                         # Mode: resume interrupted session
/inception --analyze ./my-repo              # Mode: evaluate existing repo
```

### Fresh Discovery — 5 Phases + 4 Gates

Each phase produces explicit artifacts and requires gate confirmation before advancing:

| Phase | Purpose | Output Gate | Artifact |
|---|---|---|---|
| 1. Problem space | Understand the problem deeply | Gate 1: "problem is well-defined" | none |
| 2. System scope + users | Define what the system IS and ISN'T | Gate 2: "scope confirmed" | `REQUIREMENTS.md` |
| 3. Technical decisions | Stack, auth, persistence, deploy target | Gate 3: "architecture confirmed" | `ARCHITECTURE.md` |
| 4. Development plan | Ordered phases, dependencies, DoD | Gate 4: "plan confirmed" | `PLAN.md` |
| Bootstrap | Clone template, install stack, generate CLAUDE.md | — | `CLAUDE.md` + `.claude/` configured |

**Key insight**: if the user can't pass a gate, the agent re-frames the question. Design is not closed until explicitly confirmed.

### Challenge System — 3 Levels

Risky decisions trigger graduated challenges:

- **Level 1 (Nota)**: Low-risk. Comment and continue.
- **Level 2 (Trade-off)**: Medium-risk. Explain cost, request explicit confirmation.
- **Level 3 (Bloqueo soft)**: High-risk. Require justification before advancing. Any justification accepted → register in OPEN_QUESTIONS.md → advance.

### Portability Question (Phase 2)

The agent asks: *"Of the modules you described (auth, audit, etc.) — do any need to live in another project later, or are all exclusive to this system?"*

The answer determines architecture:
- **Response A** (all exclusive) → Screaming + ports only where needed
- **Response B** (some reusable) → lib-grade structure for portable modules
- **Response C** (unsure) → Level 2 challenge; recommends ports by default; registers decision in OPEN_QUESTIONS.md

## Incremental Adoption: Yes, by Design

**Verdict: YES, this template is explicitly designed for gradual adoption.**

The `/inception --analyze ./path` mode scans an existing repo against the 6-layer standard, detects gaps, and proposes adoption in order of impact — not a big-bang rewrite.

### 4 Adoption Paths

| Path | When to use | Effort | Time to value |
|---|---|---|---|
| **Path A: Fresh project** | Starting from scratch | Low | Immediate (discovery → code) |
| **Path B: Analyze + incremental** | Existing repo, want to adopt gradually | Medium | Per-gap (hooks first, then agents) |
| **Path C: Clone + fill CLAUDE.md manually** | Want the structure without discovery | Low | 30 min setup |
| **Path D: Use only parts** | Just want hooks, or just agents | Very low | Immediate (copy what you need) |

### Adoption Recommendation Matrix

| Project Type | Recommended Path | Notes |
|---|---|---|
| New greenfield project | Fresh discovery (Path A) | Get architecture decisions in writing before coding. |
| Existing project, well-structured | Analyze + adopt agents (Path B) | Skip hooks if you have CI/CD; add agents for guidance. |
| Existing project, no structure | Analyze + hooks first (Path B) | Establish quality gates immediately. Refactor architecture later. |
| Urgent project, no time | Clone + fill CLAUDE.md (Path C) | Skip discovery; use template structure for consistency. |
| Specific need (e.g., only agents) | Use parts (Path D) | Copy `.claude/agents/` to your `.claude/` directory. |

### Known Limitations for Incremental Adoption

Be aware of these constraints when adopting:

- **Hooks assume Python**: ruff and mypy are hardcoded. For Node/Go/Java, rewrite hooks or skip them.
- **Makefile is Python/FastAPI-specific**: not generic out of the box; customize for your stack.
- **CLAUDE.md requires manual generation** if you don't use inception.
- **TL Orchestrator expects PLAN.md**: if you skip planning, the orchestrator has less context.

## Quick Start Guide: 3 Scenarios

### Scenario A: New Project from Scratch

**Time: 5 min setup + 30–60 min discovery**

```bash
# Step 1: Install inception skill globally (one-time)
mkdir -p ~/.claude/skills/inception
curl -sL https://raw.githubusercontent.com/diegosvart/main-base-claude-code/main/skill/SKILL.md \
  -o ~/.claude/skills/inception/SKILL.md

# Step 2: From any empty directory
cd ~/projects/my-new-project
/inception "I want to build a real-time audit log system for SaaS platforms"

# Step 3: Agent guides discovery → generates artifacts → bootstraps project
# Outcome: REQUIREMENTS.md, ARCHITECTURE.md, PLAN.md, CLAUDE.md + .claude/ configured
```

### Scenario B: Existing Project Evaluating Adoption

**Time: 10 min scan + 20 min decision**

```bash
# From your project directory
/inception --analyze .

# Agent reports:
# ✅ Capa 1 — CLAUDE.md presente
# ⚠️ Capa 4 — Hooks: settings.json não encontrado
# ❌ Capa 6 — Agents: não encontrado
#
# PLAN DE ADOPCIÓN SUGERIDO:
# 1. Agregar hooks base → impacto inmediato en quality gates
# 2. Instalar TL Orchestrator → coordinación de trabajo
# 3. Refactor architecture si necesario

# Step 2: Adopt incrementally (you choose the order)
# - Copy `.claude/settings.json` from template → immediate quality gates
# - Copy `.claude/agents/tl-orchestrator.md` → guidance + phase coordination
# - Later: refactor code to follow Screaming + ports pattern
```

### Scenario C: Use Only Parts (e.g., Hooks + TL Agent)

**Time: 5 min setup**

```bash
# You don't need to clone the whole template
# Just copy what you need:

# Copy hooks for quality gates
curl https://raw.githubusercontent.com/diegosvart/main-base-claude-code/main/.claude/settings.json \
  -o .claude/settings.json

# Copy TL Orchestrator agent
mkdir -p .claude/agents
curl https://raw.githubusercontent.com/diegosvart/main-base-claude-code/main/.claude/agents/tl-orchestrator.md \
  -o .claude/agents/tl-orchestrator.md

# Outcome: Quality gates active + coordination agent available
# No discovery needed; no bootstrap; you're done
```

## Key Architectural Decisions (Summarized)

### ADR-001: CLI over MCP (Non-negotiable)

**Decision**: Always CLI (gh, bash) before MCP.

**Why**: CLI tools are universally available, scriptable, and don't require setup. MCP is for integration where CLI doesn't exist.

**Applies to**: Git operations (`gh repo clone`, `gh pr create`), filesystem operations (bash), external APIs (MCP only if no CLI tool exists).

### ADR-002: Screaming Architecture + Explicit Ports

**Decision**: Domain structure ("audit", "billing", "auth") visible at directory level. Ports (ABCs) mandatory only when a module has >1 external dependency.

**Why**: Immediate clarity on business domains; minimal overhead for simple modules (only DB); portability possible without over-engineering.

**Example**:
```
app/
├── audit/          ← screams "audit domain"
│   ├── service.py  ← imports only ports.py
│   ├── ports.py    ← ABCs for storage, cache
│   └── schemas.py
└── infra/
    └── db/         ← implements ports
```

### ADR-003: claude-mem Referenced (Not Included)

**Decision**: Template recommends [claude-mem](https://github.com/thedotmack/claude-mem) for session memory but doesn't include it.

**Why**: It's a separate plugin (AGPL-3.0 licensed); template stays MIT; users opt-in for session continuity.

**Applies to**: `/inception --resume` uses claude-mem if installed (Strategy A), falls back to disk artifacts (Strategy B).

## What Comes After?

Once bootstrapped, the workflow is:

```
/new-feature "feature name"     # Create branch + spec + test structure
<code>                          # Write code, hooks auto-format + check
/review [aspect]               # Code review focused on security, types, tests
/create-pr [target]            # Create PR via gh CLI with auto-generated description
<merge>                         # Merge to develop
/close-phase [name]            # Update PLAN.md, generate summary, prompt teardown
```

Each step coordinates with the TL Orchestrator and relevant specialists (Backend, Architecture, etc.).
