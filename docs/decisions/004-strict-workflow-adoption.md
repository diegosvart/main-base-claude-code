# ADR-004: Strict Workflow Adoption for All Changes

**Date**: 2026-03-19
**Status**: Adopted
**Context**: During development of EXECUTIVE_SUMMARY.md, a commit was made directly to `main` branch, bypassing the template's documented workflow (`/new-feature` → branch → `/commit` → `/review` → `/create-pr` → merge → `/close-phase`).

---

## Decision

**All changes to the template must follow the documented workflow**, regardless of change type (code, docs, chores):

```
/new-feature "description"  # Create branch + spec
<make changes>
/commit                     # Lint + format + test + conventional message
/review [aspect]            # Code review (security, types, coverage, architecture)
/create-pr [target]         # Create PR via gh CLI
<review + approve>
<merge>
/close-phase [name]         # Update PLAN.md, generate summary, teardown
```

This applies equally to:
- **Code changes** (features, bug fixes, refactors)
- **Documentation** (ADRs, guides, summaries)
- **Configuration** (.claude/, .mcp.json, Makefile, etc.)

## Rationale

1. **Consistency**: The template defines commands and workflows. Using them ensures the template works as designed.
2. **Auditability**: Every change is documented via PR, not hidden in commit history.
3. **Quality gates**: Hooks and `/review` catch issues before merge.
4. **Session recovery**: PLAN.md + phase summaries create a "living log" of project evolution.
5. **Dogfooding**: Changes to the template should demonstrate the template in use, not bypass it.

## When This Matters Most

- **Contributors updating the template**: Workflows visible in git history = documentation of the template's maturity.
- **Teams adopting the template**: Seeing how *the template itself* is maintained builds confidence.
- **Session continuity**: Phases and summaries help `/inception --resume` recover exact state.

## Exception

None. Even one-line doc fixes, config tweaks, and chore commits use the full workflow.

## Example: Correct Flow for EXECUTIVE_SUMMARY.md

```bash
# 1. Create feature branch
/new-feature "docs: executive summary"
# → creates branch docs/executive-summary
# → creates SPEC.md, test structure, task plan

# 2. Write the document
# (make changes to docs/EXECUTIVE_SUMMARY.md)

# 3. Commit with linting + formatting
/commit
# → runs ruff format + ruff check
# → generates conventional commit message
# → summary: "docs: add executive summary for evaluation"

# 4. Review for quality
/review "all"
# → checks security, type hints, test coverage, architecture rules

# 5. Create pull request
/create-pr main
# → creates PR via gh CLI
# → auto-populates description from commit message

# 6. Merge (after approval)
git merge --no-ff docs/executive-summary

# 7. Close phase
/close-phase "executive-summary"
# → updates PLAN.md with completion status
# → generates phase-summary.md
# → lists mounted tools, prompts for cleanup
```

## How This Enables Living Documentation

By following the workflow strictly:

- **PR descriptions** become change logs (link them to issues/milestones)
- **Phase summaries** (generated by `/close-phase`) create evolution markers
- **PLAN.md** tracks what's done vs. in-progress vs. planned
- **Commits** follow conventional format → parseable by tools
- **Hooks log sessions** → audit trail of work

Tools can then consume these artifacts:
- Parse commits → timeline visualization
- Parse PRs → impact graph (what changed, why, who reviewed)
- Parse PLAN.md → Gantt charts, burndown, completion trends
- Parse phase-summaries → narrative of evolution
